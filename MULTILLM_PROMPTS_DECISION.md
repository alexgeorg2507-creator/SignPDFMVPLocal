# SignFinder — Архитектурное решение: промпты для Multi-LLM

## Решение (зафиксировано)

**Промпты остаются УНИВЕРСАЛЬНЫМИ — один текст на все 4 провайдера.**
Различия провайдеров изолированы на уровне JSON-механики, не семантики промпта.

---

## Текущая архитектура (правильная)

| Слой | Файл | Что делает |
|------|------|-----------|
| Текст промпта | `signfinder/prompts/*.py` | Единый провайдеро-независимый текст |
| JSON-механика | `signfinder/llm/base.py` `complete_structured()` | Провайдеры переопределяют нативными методами |
| Anthropic | `anthropic_client.py` | — |
| OpenAI | `openai_client.py` | function calling для JSON |
| DeepSeek | `deepseek_client.py` | response_format для JSON |
| Gemini | `gemini_client.py` | response_mime_type |

5 промптов: extraction, regex_generation, validation, party_resolver, settings.

---

## Почему универсальные, а не per-provider

1. **Стоимость поддержки**: 5 промптов × 4 провайдера = 20 текстов. Любое
   изменение бизнес-логики — правка в 20 местах, рассинхрон неизбежен.

2. **Современные модели сошлись по prompt-following**: Claude, GPT-4o, Gemini,
   DeepSeek понимают одинаковый структурированный промпт (роль → данные →
   задача → формат). Эпоха провайдеро-специфичных заклинаний прошла.

3. **Различия — на уровне механики, не семантики**: JSON-вывод уже изолирован
   в клиентах. Лимиты токенов — параметр. Семантика задачи одинакова.

---

## Известные риски (проверять на тестах)

- **DeepSeek/Gemini на русском юридическом**: сложная шапка («ООО в лице
  Генерального директора, действующего на основании Устава») — Gemini может
  путать роли. Решается примерами в промпте, не отдельным промптом.
- **Строгость JSON**: слабая модель + универсальный промпт = больше невалидного
  JSON. Решается retry-логикой и валидацией, не разным промптом.

---

## План проверки гипотезы (трёхэтапный)

**Этап 1 (сейчас, MVP):** универсальные промпты. Не плодить 20 текстов.

**Этап 2 (v1.15 автотесты):** эталонный корпус через все 4 провайдера, замерить per-provider:
- precision/recall определения нашей стороны
- доля валидного JSON с первой попытки
- точность регексов

**Этап 3 (по результатам тестов):** точечные оверрайды ТОЛЬКО где тесты показали
провал. Не «промпт на провайдера», а fallback-блок:
```
get_prompt(name) → базовый универсальный текст
  + provider_hint(name, provider) → "" или доп. инструкция
```
Большинство provider_hint вернут пустую строку. Один-два — добавку (напр. Gemini по кириллице).

---

## Следствие для roadmap

**v1.15 (автотесты) делать ДО любых правок промптов.** Без эталонного корпуса
невозможно отличить «плохой промпт» от «плохой модели» от «плохого документа».
Решение по оверрайдам — на основе измерений, не интуиции.
