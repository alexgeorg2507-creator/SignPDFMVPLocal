# RELEASE_STRATEGY.md — стратегия релизов SignFinder

Зафиксировано: июнь 2026, после завершения MVP + v1.18.0 (XOAUTH2).
**ПЕРЕСМОТРЕНО:** выбран полный разрез версий (Вариант Б), не общие зависимости.

---

## РЕШЕНИЕ (принято)

Две версии развиваются **полностью независимо, БЕЗ зависимостей между собой**.
Это НЕ «две в одном» / НЕ общий движок.

1. **Локалка 1.18** — самодостаточный репозиторий. core и api **вкомпилированы
   внутрь (vendoring)** физическими копиями. Ни от чего внешнего не зависит.
   Заморожена. Отдаётся клиенту.

2. **Облако** — отдельный репозиторий. Стартует со **свежего снапшота** core/api
   как точки отсчёта, дальше развивается **внутри облачного репо независимо**.
   Ноль связи с локалкой.

Принцип: **два отдельных дерева кода.** Фикс общего бага вносится в каждое руками
(но локалка заморожена и не чинится — минус почти не работает).

## ПОЧЕМУ ОТКАЗ ОТ ПРЕДЫДУЩЕЙ СТРАТЕГИИ

Ранее предлагалось: общий core/api как pip-зависимости с GitHub (@v1.18.0 для
локалки, @main для облака). Это «один движок, две оболочки» — СВЯЗАННЫЕ версии.
Тронул core ради облака → риск задеть локалку. Пользователю это НЕ нужно.

Отменены: переключение Dockerfile на `pip install git+...@тег`, общий core/api.

---

## ВАЖНО: текущая сборка УЖЕ вендорит

Текущие Dockerfile (`api/`, `agent/`, `streamlit/`) делают:
```dockerfile
COPY signfinder-core/ /tmp/signfinder-core/
RUN pip install --no-cache-dir /tmp/signfinder-core/
COPY signfinder-api/app/ ./app/
```
Это уже КОПИРОВАНИЕ из соседних папок, не git-зависимость. Инфра по факту ближе
к вендорингу. Нужно лишь сделать локалку самодостаточной (core/api ВНУТРИ её репо,
а не в соседних папках C:\work\).

## ЦЕЛЕВАЯ КАРТИНА

```
ЛОКАЛКА 1.18 (заморожена, отдаётся клиенту):
  signfinder-local/                  ← один самодостаточный GitHub repo
  ├── vendor/signfinder-core/        ← КОПИЯ core на момент 1.18.0
  ├── vendor/signfinder-api/         ← КОПИЯ api на момент 1.18.0
  ├── agent/ streamlit/ api/Dockerfile
  ├── docker-compose.yml
  └── (Dockerfile COPY из vendor/, контекст = сам репо)
  тег: v1.18.0-local

ОБЛАКО (своя жизнь, с нуля от снапшота):
  signfinder-cloud/                  ← отдельный GitHub repo
  ├── core/                          ← снапшот core, развивается тут
  ├── api/                           ← снапшот api, развивается тут
  └── (мультитенант, биллинг, GCS, сайт)
  НИКАК не связан с локалкой
```

## ШАГИ (после сохранности на GitHub)

### Этап 0 — Сохранность (СНАЧАЛА, отдельный промпт TASK_infra_safety_github.md)
Загнать api и local на GitHub как есть (текущее состояние, без заморозки).
Чтобы код перестал жить только на диске. ЭТО ПЕРВЫМ.

### Этап 1 — Заморозка локалки (вендоринг)
1. В `signfinder-local` создать `vendor/` (или `libs/`)
2. Скопировать снапшот `signfinder-core/` и `signfinder-api/` ВНУТРЬ репо локалки
3. Dockerfile: `COPY vendor/signfinder-core/` (из своего репо, не из ../)
4. docker-compose context — сузить до самого репо локалки (не `..`)
5. Тест: клонировать в чистую папку ВНЕ C:\work\ → собрать → работает без соседей
6. Тег `v1.18.0-local`. Заморожено.

### Этап 2 — Облако (отдельно, когда дойдём)
1. Новый repo `signfinder-cloud`
2. Скопировать снапшот core/api как стартовую точку
3. Развивать внутри: мультитенант (tenant_id), GCSStorage, биллинг, сайт
4. TD-06 закрыть (KMS/Secret Manager) — на облаке блокер
5. Локалку НЕ трогать

## СУДЬБА РЕПОЗИТОРИЕВ signfinder-core / signfinder-api

После разреза отдельные репо `signfinder-core` и `signfinder-api` как
самостоятельные единицы БОЛЬШЕ НЕ НУЖНЫ (никто не тянет их как зависимость).
Они становятся «исходным донором снапшотов»:
- можно оставить как архив-источник (где взять снапшот для облака)
- можно заморозить тегом v1.18.0 и не трогать
Их код физически переезжает ВНУТРЬ локалки (vendor/) и облака (core/api/).

Решить на этапе 1: оставить core/api репо как архив или влить в локалку и закрыть.
