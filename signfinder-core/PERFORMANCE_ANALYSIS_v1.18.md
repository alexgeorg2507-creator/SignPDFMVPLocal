# Анализ производительности v1.18 (для концепции v2.1)

Зафиксировано: июнь 2026, после серии v1.18.3–v1.18.10 (профили + dual-column + PL/MK).
Источник: разбор горячего пути по коду `signfinder/__init__.py::analyze()`,
`pdf/parser.py`, `pdf/language.py`, `pipeline/settings.py`.

---

## Симптомы (измерения клиента)

| Сценарий | До v1.18 | После v1.18.10 | Замедление |
|----------|----------|----------------|-----------|
| Применение шаблона (Amendment, 4 стр) | ~1.2с | 4777 мс | **×4** |

## Что добавилось в горячий путь после v1.18.3+

### Изменение А — Гутер-детектор колонок (parser.py, v1.18.7)

Для каждой страницы:
- `_detect_gutter`: 36 cut-позиций × N слов = O(N) проверок пересечения
- Если гутер найден: дополнительная сортировка слов + сборка двух колонок построчно
- Стоимость: 30-100 мс на страницу

### Изменение Б — Каскад langdetect (parser.py, v1.18.7)

Для каждой dual-страницы вызывается `langdetect.detect()` **дважды** (левая + правая
колонка), плюс один раз на полном документе.

```
Agreement (6 страниц, все dual) = 6×2 + 1 = 13 вызовов langdetect
До v1.18.7: 1 вызов на документ
```

Первый вызов langdetect в процессе — ленивая загрузка профилей всех языков (~300-500 мс).
Каждый последующий — 20-100 мс.

**Совокупный вклад в Agreement: ~0.7-1.5 с к каждому analyze().**

### Изменение В — detect_language с LLM fallback (language.py, существовал, но усугубился)

```python
parser_lang = doc.language[:2]
if parser_lang in ("ru","en","pl","mk"):
    return parser_lang
# иначе → полноценный LLM-вызов на определение языка
```

**Триггер для нового корпуса:** langdetect на двуязычной каше mk+en часто возвращает
`bg` (болгарский), `sr` (сербский), `hr` (хорватский) — это НЕ в списке поддерживаемых
→ срабатывает LLM fallback.

**Стоимость LLM-вызова на определение языка: 1.5-3 с.**

Это происходит **даже в шаблонном пути** (быстром пути, который раньше не вызывал LLM
вообще). Объясняет почему Addendum 1 страница занимает 3.7 с без LLM-pipeline.

### Изменение Г — detect_signer_profile (Модель Б, v1.18.3)

`detect_signer_profile`: `list_prefix("signers/") + read_json` для каждого профиля
+ подсчёт совпадений match_markers в тексте.

Стоимость: ~10-30 мс на analyze(). **Незначительно.**

### Изменение Д — Step4 LLM генерирует больше паттернов (v1.18.7b/v1.18.8)

Для dual-column документа в `markers_block` мёржатся маркеры всех языков:
- `mk`: 7 marker_words
- `en`: 4 marker_words
- Итого: 11 marker_words в промпт вместо 4

LLM пропорционально генерирует больше паттернов: 31-46 вместо ~15.

Время LLM пропорционально **выходным токенам** (декодирование) → step4 примерно ×2.

**Вклад в полный пайплайн: +1.5-3 с.**

### Изменение Е — Step3 промпт удвоился (для dual-column)

Шапка двуязычного документа = обе колонки в промпте → больше input-токенов.

Современные LLM считают prefill быстро, но это +200-500 мс к step3.

### Изменение Ж — Пост-фильтр our_side (v1.18.9, v1.18.10)

`_filter_by_our_side_context`: для каждого матча — поиск позиции в тексте страницы,
проверка вхождения синонимов в 80-символьное окно. O(matches × page_text_length).

Стоимость: 50-200 мс. **Незначительно.**

## Декомпозиция замедления

### Шаблонный путь (×4 регрессия)

Старый путь: parse → fingerprint → match. **0 LLM-вызовов.**

| Компонент | До | После | Δ |
|-----------|-----|-------|---|
| parse_pdf_bytes (1 стр) | ~100 мс | ~250 мс | +150 мс (langdetect×2 + гутер) |
| LLM detect_language | 0 | 2000-3000 мс | **+2500 мс** |
| detect_signer_profile | 0 | 20 мс | +20 мс |
| fingerprint + match | ~600 мс | ~600 мс | 0 |
| **Итого (1 стр)** | **~800 мс** | **~3700 мс** | **×4.5** |

**Главный виновник: H2 — LLM detect_language в шаблонном пути.** Срабатывает потому что
двуязычная каша → langdetect возвращает не-supported код → fallback на LLM.

### Полный путь (+50% регрессия)

| Компонент | До | После | Δ |
|-----------|-----|-------|---|
| parse_pdf_bytes (6 стр) | ~300 мс | ~1500 мс | +1.2 с |
| LLM detect_language | 0-1× | 1× | +2-3 с (часто срабатывает) |
| Step3 prompt size | базово | ×2 input | +0.3 с |
| Step4 паттерны | ~15 шт | ~40 шт | +1.5-3 с (LLM output ×2) |
| Step5 regex matching | базово | +1.5× | +0.5 с |
| Post-filter | 0 | 100 мс | +0.1 с |
| **Итого** | **~10 с** | **~15 с** | **+50%** |

## План оптимизации (v1.19)

### Приоритет 1 — убрать LLM detect_language в шаблонном пути

Шаблонному пути язык нужен только как метка (для load_template, matching).
langdetect-кода (даже `bg`/`hr`) достаточно — шаблон матчится по fingerprint, не по языку.

**Решение:** перенести LLM fallback из `detect_language()` в `run_pipeline_auto_1()`.
То есть: если langdetect дал валидный код в (ru,en,pl,mk) — берём. Если нет — для
шаблонного пути НЕ зовём LLM, используем код как есть (для matcher это не критично).
LLM-fallback включается только при полном анализе (когда уже нужна точная разметка
для extraction).

**Ожидаемый выигрыш:** шаблонный путь возвращается к ~1.2 с (×4 регрессия исчезает).

### Приоритет 2 — кэшировать langdetect внутри parse_pdf_bytes

Сейчас langdetect зовётся 2N+1 раз (N=число dual-страниц). Достаточно:
- 1× для полного текста документа (как раньше)
- 1× per язык колонки (взять одну dual-страницу как образец, не все)

**Ожидаемый выигрыш:** parse_pdf_bytes для 6-страничного dual-документа → ~500 мс
вместо 1500 мс.

### Приоритет 3 — кэшировать markers/profile внутри run_pipeline_auto_1

`load_markers`, `load_signer_profile_by_id` читают JSON с диска при каждом вызове
`get_markers_for_language` / `get_aliases_for_language` (3-5 раз за пайплайн).

**Решение:** кэш на уровне instance SignFinder, инвалидация по mtime файла.
Или (проще) — кэш на длительность одного analyze().

**Ожидаемый выигрыш:** -100-200 мс за пайплайн.

### Приоритет 4 — урезать step4 паттерны

Сейчас для dual-column в промпт идут 11 marker_words, LLM генерирует 40+ паттернов.
Многие — комбинации `Marker × Underline × Position` создающие большое декартово
произведение.

**Решение:** в промпте явно ограничить «не более 15 паттернов, только наиболее
вероятные для этого документа». Плюс детерминированное дедупирование паттернов
постпроцессом.

**Ожидаемый выигрыш:** step4 LLM с ~3 с до ~1.5 с.

### Приоритет 5 — параллелизовать independent шаги

Сейчас analyze() полностью линейный: parse → fingerprint → matcher → pipeline.
fingerprint и detect_signer_profile независимы — могут выполняться параллельно.

**Ожидаемый выигрыш:** -300-500 мс на analyze() через asyncio/ThreadPool.
Низкий приоритет — экономия мелкая, риск багов есть.

## Подтверждение гипотез

Гипотезы H1, H2, H5 — главные виновники. Подтверждение:

- **H2 (LLM detect_language)** подтверждается арифметикой: Addendum 1 страница =
  3703 мс. Без LLM-вызова такой документ обрабатывается за ~800 мс. Разница 2.9 с
  ≈ один LLM-вызов на определение языка.
- **H1 (langdetect-каскад)** — известная характеристика библиотеки: ленивая загрузка
  профилей при первом вызове.
- **H5 (step4 ×2 паттернов)** — видно в debug JSON: `final_patterns count: 31-46`
  на dual-column против ~15 на single.

После v1.19 эти гипотезы будут либо опровергнуты, либо подтверждены численно
через `timings_ms` в debug.
