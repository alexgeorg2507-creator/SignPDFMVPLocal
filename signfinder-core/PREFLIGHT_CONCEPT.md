# Pre-flight Check — концепция договорного ревью (для концепции v2.1, глава 12.4-бис)

Зафиксировано: июнь 2026, по итогам обсуждения roadmap после локального MVP.
Заменяет старую формулировку §12.4 (детерминированные R-01..R-07).

---

## 1. Смысл и позиционирование

Pre-flight check — это **второй контур** SignFinder: проверка самого договора
ПЕРЕД подписанием, а не поиск места подписи.

Это то что превращает продукт из утилиты («расставь факсимиле») в **договорного
агента** («проверь договор как сделал бы юрист, дай замечания, и подпиши»).

Финальное ревью договора + расстановка мест подписи — это **типовой офисный
бизнес-процесс**. SignFinder автоматизирует обе половины, а не одну.

---

## 2. Ключевое решение: LLM-first, без детерминированной базы

Старая концепция (§12.4) предполагала 7 детерминированных правил R-01..R-07
под РФ-договоры (regex/проверки наличия разделов).

**Решение пересмотрено: LLM-first для всех языков.**

Причины отказа от детерминированной базы:
- Основной корпус — PL/MK/EN, а R-01..R-07 заточены под РФ. Язык-специфичные
  regex-правила под 4 языка = работа в стол.
- LLM читает договор на любом языке из коробки. Один промптовый каркас работает
  для ru/en/pl/mk сразу.
- Замечания человеческим языком («срок в п.4.2 истёк раньше даты подписания»)
  несут ценность, детерминированные галочки («раздел Реквизиты присутствует ✓») —
  нет.

Гибрид остаётся философией: детерминированную базу можно добавить ПОЗЖЕ как
быстрый/дешёвый слой, если LLM окажется дорогим или медленным на потоке. Но
стартуем с LLM.

---

## 3. Промпт под язык и юрисдикцию (динамический)

Ключевая фича — промпт ревью **формируется под язык и юрисдикцию документа**.

```
Язык документа определён (ru/en/pl/mk) →
  выбирается юрисдикционный контекст →
    промпт «проверь договор по нормам [юрисдикция]» →
      LLM выдаёт замечания на языке документа
```

Это делает SignFinder не «проверяльщиком по чек-листу», а агентом который
понимает что договор по польскому праву проверяется иначе чем по российскому.
Динамично, показывает глубину продукта.

Юрисдикционные профили (стартовые):
- ru → РФ (ГК РФ): реквизиты, предмет, срок, расчёты, ответственность
- pl → Польша (KC): odpowiednie sekcje, strony, przedmiot, termin
- en → общий/EU: parties, subject, term, payment, liability, governing law
- mk → Северная Македония (по аналогии, уточнить по первому клиенту)

Профиль = блок текста в промпте, редактируемый через UI (как другие промпты).

---

## 4. Что проверяет ревью (универсальные оси)

Независимо от юрисдикции, LLM проверяет наличие и непротиворечивость:

| Ось | Что смотрит |
|-----|-------------|
| Стороны | Обе стороны идентифицированы, реквизиты есть |
| Предмет | Предмет договора определён однозначно |
| Сроки | Срок действия указан, не противоречит другим пунктам |
| Расчёты | Порядок и сумма расчётов присутствуют |
| Ответственность | Раздел ответственности есть |
| Подписи | Места для подписи обеих сторон присутствуют |
| Противоречия | Внутренние конфликты между пунктами |

Это НЕ жёсткий чек-лист (как старые R-01..R-07), а оси внимания для LLM.
Модель сама решает что существенно для конкретного договора.

## 5. UX: информационный режим, два светофора

### Информационный режим (НЕ блокирующий)

Pre-flight НЕ меняет договор и НЕ блокирует подпись. Выдаёт **набор соображений**
по текущей версии документа. Оператор сам решает что с ними делать.

Обоснование: клиент знает свой договор лучше системы. Блокировать подпись из-за
LLM-замечания — слишком жёстко. Прятать — бесполезно. Информировать — правильный
баланс: «вот что я заметил, решай сам».

### Два независимых светофора

Светофор подписи и светофор ревью — **разные оси**, не смешиваются:

```
🟢 Подпись          🟡 Рев低 договора
Места найдены       Есть 3 замечания
уверенно            к проверке
```

Пример комбинации: «Подпись расставлена уверенно (🟢), но в договоре есть
замечания (🟡)» — два разных факта, оба видны оператору. Объединение в один
статус потеряло бы информацию.

Светофор ревью:
- 🟢 green — замечаний нет, договор целостен
- 🟡 yellow — есть замечания (показать список)
- 🔴 red — критичные проблемы (нет сторон, нет предмета) — НЕ блок, но явный флаг

## 6. Где появляются замечания (точки вывода)

Порядок реализации — от простого к сложному:

1. **Streamlit (первый этап)** — вкладка/блок «Замечания по договору» рядом с
   результатом подписи. Это MVP вывода.
2. **Письмо агента (выход клиенту)** — в ответном письме с подписанным PDF
   добавляется блок замечаний. «Документ подписан. Обратите внимание: [замечания]».
3. **Landing + sandbox (демо)** — экран «что система заметила в договоре» как
   часть онбординга. Показывает ценность агента, не только подписывальщика.

---

## 7. Что это даёт продукту и продажам

- **Воспринимаемая ценность:** «договорный агент» вместо «утилита для факсимиле»
- **Второй крючок для разговора с клиентом:** не «подписываю», а «проверяю и
  подписываю» — звучит как помощник, не как инструмент
- **Демо-история целостна:** загрузил → проверил → нашёл места → подписал →
  получил замечания. Полный процесс, не половина.
- **Дифференциация:** конкуренты (ручные PDF-signers) этого не делают вообще

---

## 8. Объём (честная оценка)

Pre-flight с LLM-ревью — это **самостоятельная фича, сопоставимая по объёму с
поиском подписи**. Не «маленькое дополнение». Несколько недель работы.

Оправдано: это то что превращает утилиту в продукт. Но закладывать реалистично.

Версия: **v1.20** (после неё MVP функционально завершён, переход к облаку).

## 9. Связь с остальным roadmap

```
v1.20 Pre-flight (LLM-ревью, два светофора, информационный)
   ↓ MVP функционально завершён
Облако (Azure) + Landing + Sandbox
   ↓ pre-flight входит в демо
Скрининг спроса на полном процессе (ревью + подпись)
```

Pre-flight ДО облака и ДО лендинга — потому что демо должно показывать полный
бизнес-процесс, а не половину.
